# Design da publicação v0.3.0

## Objetivo

Publicar a versão 0.3.0 do Supa.cc de forma linear e verificável no GitHub,
PyPI e Homebrew, reutilizando o workflow de Trusted Publishing e o tap já
mantido no próprio repositório.

## Estado inicial

- `pyproject.toml` e `supa_cc/__init__.py` definem a versão 0.3.0.
- A branch local `main` contém 35 commits ainda não enviados e alterações não
  commitadas de preparação da release.
- A tag e a GitHub Release `v0.3.0` ainda não existem.
- O projeto ainda não possui uma distribuição publicada no PyPI.
- `.github/workflows/release.yml` publica por OIDC usando o environment `pypi`.
- `Formula/supa-cc.rb` permanece em `v0.2.0` até a publicação no PyPI ser
  concluída e verificada.
- O ambiente Linux atual não possui Homebrew; auditoria e smoke test da fórmula
  exigem um host macOS com Homebrew.
- O push direto para `main` usará o bypass administrativo autorizado, apesar da
  proteção que normalmente exige pull request e uma aprovação.

## Fluxo da release

### 1. Preparar o candidato

Revisar o histórico e todas as alterações pendentes, confirmar que pertencem à
0.3.0 e procurar segredos, caminhos locais, caches e conteúdo privado. Executar
a suíte completa, validação de dependências, auditoria, build e inspeção dos
artefatos. Instalar a wheel em ambiente descartável e confirmar os dois comandos
de versão.

Organizar as alterações pendentes em commits por responsabilidade, preservando
os 35 commits existentes. A fórmula, a data final do changelog e os textos que
afirmam disponibilidade permanecem inalterados nessa fase.

### 2. Validar e publicar o commit candidato

Enviar `main` diretamente para `origin/main` e aguardar a CI no SHA exato. A tag
só pode ser criada após a matriz obrigatória ficar verde. Criar e enviar a tag
anotada `v0.3.0` nesse SHA e publicar uma GitHub Release estável com notas
derivadas do changelog. Builds locais não serão anexados à release.

### 3. Publicar e verificar o PyPI

A publicação da GitHub Release dispara o workflow existente. O workflow valida
a correspondência entre tag e versão, testa, constrói e inspeciona os artefatos
uma única vez. O job `publish` usa apenas OIDC e o Trusted Publisher do
environment `pypi`; nenhum token manual será criado.

Os jobs de verificação instalam `supa.cc==0.3.0` diretamente do PyPI no Linux e
no Windows. A página da versão, os arquivos publicados, seus metadados e uma
instalação independente por pipx também serão conferidos antes de prosseguir.

### 4. Publicar e verificar o Homebrew

Somente após a verificação do PyPI, baixar o tarball real da tag e calcular seu
SHA256. Atualizar a URL, o checksum e os recursos Python da fórmula. Em um host
macOS com Homebrew, executar atualização de recursos, auditoria estrita,
instalação a partir do código-fonte e teste da fórmula.

Commitar e enviar a fórmula verificada para `main`, que é também a fonte do tap
`dgabreuu/supa-cc`. Fazer uma instalação limpa pelo tap e confirmar a versão
0.3.0. Se não houver um host macOS adequado, o processo para nesse ponto sem
publicar uma fórmula não validada.

### 5. Finalizar a documentação

Depois de PyPI e Homebrew estarem disponíveis, registrar a data real no
changelog, trocar o limite de comparação `HEAD` por `v0.3.0` e remover os avisos
de canais ainda planejados do README e do guia de instalação. Enviar esse commit
final e aguardar sua CI.

## Responsabilidades e proveniência

- Git e GitHub determinam o commit e a tag imutáveis da release.
- O GitHub Actions constrói e publica os artefatos Python a partir da tag.
- O PyPI Trusted Publisher autentica a publicação sem segredo persistente.
- A fórmula Homebrew referencia exclusivamente o tarball real da tag publicada.
- Commits pós-tag contêm apenas a fórmula e a finalização documental; eles não
  alteram o conteúdo Python já publicado como 0.3.0.

## Tratamento de falhas

O processo deve parar antes da próxima etapa irreversível quando houver:

- divergência de versão ou de SHA;
- segredo, caminho privado ou arquivo indevido;
- teste, build, auditoria ou inspeção com falha;
- CI incompleta ou falhando no commit candidato;
- tag ou versão remota preexistente inesperadamente;
- falha de permissão, OIDC ou Trusted Publishing;
- artefato ou checksum não reproduzível;
- ausência de host macOS capaz de validar a fórmula.

Uma versão já publicada no PyPI não será sobrescrita nem recriada. Falhas após
essa publicação exigem correção compatível com a imutabilidade dos artefatos e,
quando necessário, uma nova versão.

## Critérios de conclusão

- A tag anotada e a GitHub Release `v0.3.0` apontam para o candidato validado.
- O workflow de release conclui build, publicação e verificação com sucesso.
- `supa.cc==0.3.0` pode ser instalado do PyPI e reporta 0.3.0.
- O tap instala e testa a fórmula `v0.3.0` em macOS.
- Changelog e documentação descrevem corretamente a disponibilidade real.
- `main` remoto contém os commits finais e não há segredo ou artefato indevido.
